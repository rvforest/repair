# Design: Replace Zellij Integration With Shell Hooks

## Context
The current implementation reads pane scrollback from Zellij and parses the last command from visible terminal output. That approach is coupled to one terminal multiplexer, forces users into a specific runtime, and still depends on brittle prompt parsing. Shell hooks are a better fit because they run at the moment a command starts and completes, which gives the tool direct access to the executed command, exit status, and captured output.

## Goals / Non-Goals

**Goals:**
- Remove the Zellij runtime dependency entirely
- Capture the most recent command session in a shell-agnostic storage format under the XDG state directory
- Provide a guided setup flow through `repair init <shell>`
- Keep the capture writer inside the TypeScript CLI so the shell snippets stay thin and do not depend on `jq`
- Preserve the existing LLM analysis, redaction, caching, and output formatting pipeline

**Non-Goals:**
- Support every shell equally in the first pass if full output capture is not technically reliable
- Capture full shell history or multiplexed pane history
- Automatically modify user shell rc files during install
- Keep backward compatibility with Zellij-based capture

## Decisions

### Decision: Use shell integration snippets generated by the CLI
**Why:** This matches the common CLI pattern used by tools like Starship and zoxide. It keeps setup explicit, avoids writing to dotfiles automatically, and lets the integration evolve with the installed binary.

**Details:**
- `repair init zsh` and `repair init bash` print shell code to stdout
- Users add `eval "$(repair init zsh)"` or `eval "$(repair init bash)"` to their shell config
- Unsupported or partial-support shells should receive a clear error or limited-mode message instead of silent degradation

### Decision: Shell snippets call an internal CLI subcommand
**Why:** Writing JSON in shell is awkward and would otherwise require external tools. Moving serialization into TypeScript keeps the snippets short and centralizes validation.

**Internal commands:**
- `repair _write-session --cmd ... --output ... --code ... --ts ...`

### Decision: Support robust output capture where shell hooks can guarantee it
**Why:** Zsh and Bash can intercept pre-command and pre-prompt boundaries well enough for an MVP. Fish and Nushell have different hook semantics and should not be spec-required unless the implementation can guarantee equivalent output capture.

**MVP compatibility:**
- Required: `zsh`, `bash`
- Deferred: `fish`, `nushell`

### Decision: Replace environment detection with session availability detection
**Why:** The CLI should fail based on whether capture data exists, not whether it is running inside a particular terminal.

**New runtime flow:**
1. Load config
2. Read the most recent captured shell session
3. Error if shell integration is not installed or no captured session is currently available to analyze
4. Redact secrets, consult cache, call LLM, and print formatted output

## Architecture

```text
┌───────────────────────┐
│ User shell            │
│ preexec + precmd hook │
└───────────┬───────────┘
            │ captures command, output, exit code
            v
    ┌──────┴────────┐
    │               │
    v               v
┌───────────────────────────────┐
│ repair _write-session         │
│ writes session JSON           │
└───────────────┬───────────────┘
                │
                v
┌──────────────────────────────────────────┐
│ repair CLI                               │
│ 1. load config                           │
│ 2. read last-session.json if present     │
│ 3. redact secrets / confirm / cache      │
│ 4. send command + output to LLM          │
│ 5. render explanation and fixes          │
└──────────────────────────────────────────┘
```

## Data Flow

1. User runs a command in a configured shell
2. The shell preexec hook records the command text and starts output capture
3. The shell precmd hook restores terminal output and captures the exit code
4. The shell invokes `repair _write-session` and stores structured session JSON in the state directory
5. When the user runs `repair`, the CLI reads the JSON and builds the analysis request if session data is available
6. Existing security filtering, caching, and LLM provider code runs unchanged apart from richer shell metadata

## Risks / Trade-offs

### Risk: Shell hook output capture can interfere with prompts or redirections
**Mitigation:** Keep shell snippets minimal, document supported shells explicitly, and test against common prompt frameworks.

### Risk: Captured output can be large
**Mitigation:** Keep truncation in the analysis pipeline and consider optional limits in the writer if memory becomes a concern.

### Risk: Users upgrade from Zellij-based workflow and lose setup
**Mitigation:** Replace Zellij errors and docs with a concrete migration message that points users to `repair init <shell>`.

### Trade-off: Breadth of shell support vs reliability
**Decision:** Require reliable capture for Bash and Zsh first rather than shipping weaker support claims across every shell.

## Migration Plan

1. Remove Zellij-specific code paths and error messages
2. Add shell integration generation and state file reader/writer modules
3. Update docs and help text to instruct users to enable shell hooks and explain how shell capture works
4. Replace the Zellij spec capability with shell hook requirements and mark the old capability removed

## Open Questions

1. Should the initial implementation expose `repair init fish` as unsupported, or ship a command-only integration that does not capture output?
2. Should the session writer overwrite a single file, or keep a short rolling history for future commands like `repair last --nth 2`?
3. Should shell integration include current working directory capture for better analysis context in multi-repo workflows?
